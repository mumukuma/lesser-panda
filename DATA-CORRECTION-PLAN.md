# 資料校訂機制 — 規劃文件

> 對應 ROADMAP #12（資料修正機制）。本文是規劃草案，尚未實作。
> 建立：2026-06-15

## 目標

讓「熟悉的熊友」能回報兩類資訊，由站長校訂後寫回 wiki：

1. **修正既有資料**：欄位錯了（名字、生卒、性別、居住史…）或**家族關聯錯了**（父母／雙胞胎／子女關係）。
2. **補新事件**：個體**出生**（新生兒建檔）與**死亡**（補卒日、加 🌈）。

站長一個人無法逐筆對照，因此用**友善表單**讓熊友提案；初期站長自審，後期開放信任的網友協審。

## 最高原則（呼應 CLAUDE.md，不可違反）

- **表單 = 提案，`wiki/*.md` = 唯一正本。** 任何提交都**不會自動寫進 wiki**，一律經站長核可後才動。
- **來源必填且須可驗證**：動物園公告／官網／新聞／IG 貼文連結／照片。沒有可信來源的提案直接退回或標「需補件」。
- **衝突一律以 wiki（站長校訂）為準**，表單只負責「提示站長檢視」，不得用來覆蓋既有校訂。
- **每次套用都要寫 `wiki/log.md`**（記為 `fix` 或 `update`／`add`），保留變更軌跡。

## 整體架構（無後端，配合靜態站）

網站是 Astro + GitHub Actions 的純靜態站，沒有伺服器可收表單。因此用「第三方表單 → 試算表佇列 → 站長審核 → 手動／Claude 協助寫回 wiki」這條路徑：

```
熊友 ──填──▶ 友善表單 ──自動──▶ 試算表（審核佇列，含「狀態」欄）
                                        │
                              站長（+未來協審者）審核
                                        │
                          核可 ──▶ 更新 wiki/*.md（依 SCHEMA）
                                        │
                       build_db.py → audit.py → export_json.py
                                        │
                          git commit → GitHub Actions 自動部署
```

### 表單工具建議

| 工具 | 優點 | 缺點 |
|---|---|---|
| **Tally**（推薦） | 免費、條件式邏輯（依「類型」顯示不同欄位）、可嵌入 Astro 站、填表人免註冊、外觀可調（之後配合 UI 童趣化） | 需自建帳號管理 |
| Google 表單 | 所有人都會用、最穩、自動進 Google Sheet | 條件邏輯弱、外觀陽春 |

兩者都能把回應自動匯到試算表。**建議用 Tally**：因為要做兩種型態（修正 vs 事件），條件式邏輯能讓填表人只看到該填的欄位；且未來可嵌進網站、配合童趣風格。若想最快上線，Google 表單也完全可行。

## 表單設計（分開兩張）

「修正」與「事件」欄位差很多，分開設計，填表者清楚、審核也快。

### 表單 A — 資料修正表

| 欄位 | 必填 | 說明 |
|---|---|---|
| 要修正的個體 | ✅ | 名字 + RPF 連結或 wiki 頁面連結（盡量貼連結，避免同名混淆） |
| 錯誤類型 | ✅ | 多選：中文名／日文名／暱稱·別名／生日／卒日／性別／居住史／**家族關係**／其他 |
| 目前 wiki 寫的是 | ✅ | 現值（讓站長快速定位） |
| 正確應該是 | ✅ | 建議的新值 |
| 來源／證據 | ✅ | 連結或說明（動物園公告、新聞、IG、官網、照片） |
| 補充說明 | — | 自由填寫 |
| 你的稱呼／聯絡（選填） | — | 想回覆結果時用 |

> **家族關係修正特別注意**：關聯錯誤（如父母標錯、雙胞胎漏列）是**最高成本**的修正，因為要同時改雙方頁面、改 wikilink、重建 DB。表單可在「家族關係」被勾選時，多問一題「是哪一種關係？涉及哪幾隻？」（Tally 條件式）。

### 表單 B — 出生／死亡事件表

第一題「事件類型」分流：

**出生（新生兒）**

| 欄位 | 必填 | 說明 |
|---|---|---|
| 寶寶名字（若已命名） | — | 未命名可留空，註明「未命名」 |
| 生日 | ✅ | ISO，只知年份就填年份 |
| 出生動物園 | ✅ | 用日文原名 |
| 父親 / 母親 | ✅ | 名字或連結（建檔要補直系親屬） |
| 是否雙／多胞胎 | — | 若是，列出手足 |
| 來源 | ✅ | 動物園公告／新聞連結 |

**死亡**

| 欄位 | 必填 | 說明 |
|---|---|---|
| 過世的個體 | ✅ | 名字 + 連結 |
| 卒日 | ✅ | ISO |
| 最後居住動物園 | — | 若有遷移 |
| 來源 | ✅ | 公告／新聞連結 |

> 死亡事件套用時：更新該頁 `died:`、內文與相關頁面手足／子女標註加 🌈。

## 審核流程（intake → review → apply）

### 1. 收件（intake）
表單回應自動進試算表。在試算表加一欄 **狀態**：`待審 / 需補件 / 已核可 / 已套用 / 已退回`，當作站長的審核佇列。

### 2. 審核（review）— 站長
逐列檢查：

- 來源是否可信、可驗證？無來源 → 標「需補件」並（若留聯絡）請補。
- 是否與現有 wiki 衝突？**衝突以 wiki 為準**，除非提案來源明確更可靠，否則保留 wiki。
- 家族關係類：影響範圍多大（涉及幾隻）？標記為高成本待辦。
- 判定後更新「狀態」欄。

### 3. 套用（apply）— 站長 / Claude 協助
核可的項目依 `SCHEMA.md` 更新 `wiki/*.md`。可把核可的試算表列直接貼給 Claude，請它依規範產生 Edit（含自動補直系親屬、回頭把純文字親屬改 wikilink）。流程同 CLAUDE.md「新增成員流程」：

```
改 wiki → python3 tools/build_db.py → python tools/audit.py
        → python3 site/scripts/export_json.py → 在 log.md append 一筆 → commit
```

`tools/audit.py`（與 lineage 比對）可在套用後當 sanity check。

### 4. 回覆（選填）
若填表人留了聯絡方式，告知「已採用／未採用＋原因」，鼓勵持續回報。

## 演進路徑（含未來網友協審）

**階段 0｜現況**：無機制，站長手動發現與修正。

**階段 1｜MVP（先做這個）**
友善表單（A + B）→ 試算表佇列 → 站長獨審 → Claude 協助寫回 wiki。先求跑得動，把回報管道打開。

**階段 2｜協審者（信任的少數熊友）**
無後端下最務實的做法：

- **試算表協審法**（推薦，對非技術熊友友善）：給 2–3 位信任熊友試算表的「**評論／檢視**」權限，新增「協審意見」「可信度（高/中/低）」欄。他們幫忙查證、補來源、標可信度，**但最終仍由站長核可**（雙簽：協審看過 + 站長核可）。
- GitHub Issue 協審法（僅適合會用 GitHub 的協審者）：把待審項目開成 Issue，協審者留言／投票。因為協審者只是少數人，這條路可行，但一般熊友走表單即可，不必碰 GitHub。

**階段 3｜信任分級**
協審者累積可靠紀錄後，其意見權重提高；可訂規則如「2 位協審查證一致 → 站長快速核可」。**但仍不開放自動寫入** wiki，資料主權永遠在站長。

> 演進核心：開放的是「**查證與提案的協作**」，不是「寫入權」。寫入 wiki 永遠經站長最終核可。

## 建議的下一步（按順序）

1. 決定表單工具（Tally vs Google 表單）。
2. 把表單 A、B 的欄位定稿、建出表單、接上試算表、加「狀態」欄。
3. 在網站放一個明顯的「回報 / 補資料」入口（連到表單）。
4. 小規模試跑：找 1–2 位熊友丟幾筆真實回報，走完一輪「收件 → 審 → 寫回 → log」，修正流程摩擦點。
5. 順手後，再規劃階段 2 的協審者名單與試算表權限。

需要時，Claude 可協助：產生表單欄位文案、寫網站上的回報入口、把核可的試算表列批次轉成 wiki 條目／Edit。
