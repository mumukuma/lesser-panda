# 批次 2 — 「轉園」回報處理 SOP

> 收到一筆「🚚 轉園（換動物園）」回報後，照這份做。
> 一切遵循 `CLAUDE.md`：`wiki/*.md` 是唯一正本，回報只是提示，校訂後才寫入。
> 動物園名以 `data/zoos.json`（註冊表）為唯一事實來源。

---

## 這類回報長什麼樣

從個體頁「✏️ 回報這頁資料／近況」按鈕來，收件匣會有 `panda`／`slug`／`url` 三欄帶入身分。表單第 1 題是**下拉選單**並用條件邏輯分流：選「🚚 轉園（換動物園）」時，只會顯示轉園相關欄位（「正確的應該是什麼？」會被隱藏）：

- **到哪一園**（轉園時**必填**）— 常是簡稱，例：只寫 `tama`、`東山`
- **生效日期**（選填，轉園才填）— 可能精確到日，也可能只給月份
- **來源**（官網／公告／新聞／IG／現場親眼所見＋日期）

> 因為條件分流，轉園回報「只需填新園」即可送出，不會被更正類的必填描述卡住。
> 「從哪一園」不問——我們已知該個體現居園，從 wiki 取即可。

---

## 步驟 0：定位是哪一隻

用收件匣的 **`slug`** 欄直接定位 `wiki/<slug>.md`（最精準）；`url` 可點開核對。少數沒帶參數的（直接開 `/feedback`）就靠描述名字，用 `python tools/query.py profile <名>` 找，同名多隻再靠生日／現居園縮小。

## 步驟 1：紅綠燈分級

| 燈號 | 條件 | 動作 |
|------|------|------|
| 🟢 綠 | 有可信來源（官網／官方社群／新聞／可查證 RPF）**且**有明確生效日期 | 進步驟 2 |
| 🟡 黃 | 有來源但**日期不明／模糊**，或來源是「我記得／聽說／沒附連結」，或簡稱對不準 | 補查官網／RPF 後再升🟢或降🔴；查不到就暫不改 |
| 🔴 紅 | 無任何來源，或明顯灌水／與多個可靠來源矛盾 | 收件匣標「退回＋原因」，**不動 wiki** |

> 轉園的關鍵是**生效日期**：居住史表是時間軸，日期錯了整條時間軸就錯。日期不明寧可維持🟡，回問來源或用官網公告日。

## 步驟 2：解析「到哪一園」的簡稱 ★本批重點

回報常只寫簡稱（`tama`／`東山`／`nasu`）。用解析工具對到註冊表 canonical，**不要用記憶硬猜**：

```bash
python tools/resolve_zoo.py tama
```

- **✅ 精準命中** → 直接用它回的 canonical。
- **🟡 單一候選** → 核對地點無誤後採用；順手把這個簡稱補進該園別名（下次就精準）：
  ```bash
  python tools/resolve_zoo.py --add tama 多摩動物公園
  ```
- **🟡 多個候選** → 對照來源（官網／地點）挑一個，別猜。
- **🔴 查無此園** → 八成是**還沒登記的新園**，走步驟 3。

> 簡稱解析不確定時，這筆降🟡，回問來源或回報者，別賭。

## 步驟 3：新園若沒登記，先補 `data/zoos.json` ⚠️ 否則 build 中止

`build_db.py` 對園名做**精確比對**，寫了註冊表沒有的名會**報錯中止**。所以新園要先登記一筆：

1. 確認真的沒有：`python tools/resolve_zoo.py <新園名>` 回🔴，或翻 `data/zoos.json`。
2. 在 `data/zoos.json` 加一筆（陣列尾端 append 一個物件），欄位：
   - `canonical`：**正式全名**，日本園用日文全名（如「○○市□□動物園」），對外顯示用這個
   - `lineage_id`：有對應就填，沒有填 `null`
   - `en`、`zh`：英／中名（沒有填 `null`）
   - `country`、`location_ja`、`location_en`：國別與所在地
   - `lat`、`lng`：座標（查官網／Google 地圖）
   - `map`、`website`、`logo`：地圖連結、官網、logo（沒有填 `null`）
   - `aliases`：`[]`，把這次回報用的簡稱也放進去（如 `["tama"]`）
3. 存檔後 `python tools/resolve_zoo.py <新園名>` 應變成 ✅ 精準命中。

> lineage 僅用來初次帶入座標，非權威；座標以官網／地圖為準。

## 步驟 4：改 frontmatter `zoos:`（唯一居住史來源）

居住史只改 frontmatter，**內文「## 居住史」表格自動生成、絕不手改**。格式 `園名 (起 – 訖)`，破折號用 `–`，現居訖日寫「現在」。

轉園 = 把舊園（現居那行）的訖日填上、再加一行新園：

```yaml
# 改前
zoos:
  - 多摩動物公園 (2006-06-22 – 現在)

# 改後（2026-05-06 轉到長野市茶臼山動物園）
zoos:
  - 多摩動物公園 (2006-06-22 – 2026-05-06)
  - 長野市茶臼山動物園 (2026-05-06 – 現在)
```

- **離開日＝抵達日＝生效日**（同一天，與既有條目慣例一致）。
- 新園那行寫**步驟 2／3 解析出的 canonical**，不是回報的簡稱。
- 日期只給月份就只寫到月（如 `2026-05`），別自己補日。

## 步驟 5：重建管線

```bash
python3 tools/gen_residence.py    # 依新的 zoos: 重生內文「## 居住史」表
python3 tools/build_db.py         # 重建 DB（園名未登記會在這報錯）
python3 site/scripts/export_json.py   # 重生網站資料
```

`build_db.py` 沒報錯 = 園名都對得上註冊表。若報「未登記」就回步驟 3。

## 步驟 6：記 log

在 `wiki/log.md` 末端 append 一筆 `update`，名字用 backtick，**絕不用 `[[wikilink]]`**：

```markdown
## [YYYY-MM-DD] update | 依回報更新 `名字` 轉園

**來源**：
- <回報附的官網／公告連結>

**更新條目**：
- `slug.md` — 居住史新增 <新園 canonical>（YYYY-MM-DD 生效），舊園訖日補上
```

新登記了園的話，這筆也提一句「`data/zoos.json` 新增 `<新園>`」。

## 步驟 7：驗收與收尾

- 開該個體頁，確認「## 居住史」表多了新園一列、「現居」指向新園。
- `python tools/audit.py` 看完整度（缺欄位／與 lineage 不符提示）。
- 收件匣（Google Sheet）把該筆標「已處理」，要致謝就記下暱稱。
- 推 `main` → GitHub Actions 自動部署。
