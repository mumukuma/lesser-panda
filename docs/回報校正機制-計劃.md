# 使用者回報校正機制 — 完整計劃

> 對應 ROADMAP #12「資料修正機制（填表 + 生卒補資料）」
> 撰寫：2026-06-19 ｜ 狀態：待你裁示

---

## 一句話總結

在網站內嵌一張「回報表單」，讓一般愛好者把錯誤或新消息丟進來；所有回報先進到一份**收件匣（試算表）**，由你（搭配 Claude 幫忙查證、起草）逐筆判斷後，才寫進 `wiki/*.md`。**表單只是輸入管道，wiki 永遠是唯一正本。**

---

## 核心原則（不可動搖）

這套機制完全服從 `CLAUDE.md` 的資料主權規範：

1. **`wiki/*.md` 是唯一正本**，使用者回報一律經你校訂後才寫入，不自動覆蓋。
2. **回報 ≠ 事實**。回報是「提示你去檢視」，不是「指令照改」。
3. **衝突時以 wiki 為準**，不因為有人回報就推翻你既有的校訂。
4. 名稱（尤其中文名、暱稱）以**有來源佐證**者為準，匿名說法只當參考。

---

## 整體流程（鳥瞰圖）

```
 一般愛好者
     │  （在網站點「回報資料」）
     ▼
 內嵌表單（Tally / Google 表單）
     │  自動寫入
     ▼
 收件匣試算表（Google Sheet）   ←── 你的「待辦清單」
     │
     ▼
 你 + Claude 逐筆判斷
     │  紅綠燈分級（見下）
     ├─🟢 採用 → Claude 起草 wiki 修改 + log → 你按「同意」→ 寫入
     ├─🟡 待查 → 補查官網/RPF/lineage 後再決定
     └─🔴 退回 → 標記原因，不動 wiki
     │
     ▼
 重建管線（gen_residence → build_db → export_json）→ 推 main → 自動部署
```

整個過程沒有任何一步是「使用者直接改到網站」。你始終是最後一道關卡。

## 一、回報管道：內嵌表單怎麼選

網站是 GitHub Pages 靜態站、沒有後端，所以不能自己收表單，要借**第三方表單服務**，再用 `<iframe>` 內嵌進 Astro 頁面。三個務實選項：

| 方案 | 內嵌 | 收件匣 | 對愛好者門檻 | 成本 | 適配度 |
|------|------|--------|------------|------|--------|
| **Tally**（推薦）| iframe，外觀乾淨可客製 | 內建後台＋可同步 Google Sheet | 低（免登入）| 免費足夠 | ★★★★★ |
| **Google 表單** | iframe（樣式較陽春）| 原生 Google Sheet | 低（免登入）| 免費 | ★★★★☆ |
| GitHub Issues + 範本 | 連結跳轉，非內嵌 | Issue 列表 | 高（要 GitHub 帳號）| 免費 | ★★☆☆☆ |

**建議：Tally。** 理由：可做「條件式分流」（選了哪類回報就只顯示該類欄位）、外觀能配合網站童趣風（呼應 ROADMAP #11）、又能同步成 Google Sheet 當收件匣。Google 表單是穩健的備案，差別只在外觀與分流彈性。

GitHub Issues 對你最省事（直接連到 repo），但會把一般愛好者擋在門外，與「一般使用者回報」的目標衝突，所以列為最後選項。

> 內嵌方式：在 `web/` 新增一個 `/feedback` 頁面（或在每個個體頁尾加「這頁資料有誤？回報」按鈕），放表單的 iframe。Tally/Google 都提供現成 embed 碼。

## 二、一張表單，分流三類回報

第一題用單選把回報分流，後續欄位依類型顯示。共用欄位先收齊「是誰、哪一隻、有沒有來源」。

**共用欄位**

- 這筆回報是關於哪一隻？（個體名字／網址，可不確定）
- 你的暱稱（選填，想被致謝可留）
- **來源／證據**（最重要）：動物園官網或公告連結、新聞、IG 貼文、RPF 連結，或「現場親眼所見＋日期」

**A. 資料更正**（關係、名字、生日等錯了）

- 哪裡錯了？（關係連結／名字／生日／物種／其他）
- 正確的應該是？（例：不是兄弟，是**雙胞胎**；生日應為 2020-06-29）

**B. 資料變更**（轉園等近況變動）

- 變動類型：轉園／其他
- 從哪一園 → 到哪一園，**生效日期**

**C. 出生 / 過世**

- 事件：新生 baby／過世
- 日期、發生在哪一園
- （新生）父母是誰？同胎幾隻？
- （過世）享年或補充

## 三、判斷準則：紅綠燈可信度系統 ★核心

這是讓你「一眼判斷」的關鍵。每筆回報先問一句話：**「有沒有可信來源？它和 wiki 既有校訂衝突嗎？」**，落到三盞燈：

### 🟢 綠燈 — 可放心採用

- 有**官方／可靠來源**（動物園官網、官方社群公告、新聞、可查證的 RPF 更新），**且**
- 只是「補空白」或「補新事件」，不推翻你既有校訂。
- 例：官網公告某隻轉園、新生兒出生公告、官網訃聞。
- 動作：Claude 起草修改，你看一眼按「同意」即可。

### 🟡 黃燈 — 先查證再決定

- 與現有 wiki **不一致**，或來源是「我記得 / 聽說 / 沒附連結」。
- 例：說某兩隻是雙胞胎但沒來源；說名字拼法不同。
- 動作：Claude 去比對 RPF／lineage／官網，補齊佐證後再升級成綠或降成紅。查不到就維持黃燈、暫不改，必要時回信問回報者要來源。

### 🔴 紅燈 — 暫不採用

- **無任何來源，且要求覆蓋你已校訂的資料**（尤其中文名、暱稱、生日）。
- 明顯惡意、灌水、與多個可靠來源矛盾。
- 動作：在收件匣標記「退回＋原因」，**不動 wiki**。這正是資料主權原則：匿名說法不能推翻作者校訂。

> 口訣：**綠燈補空白、黃燈先查證、紅燈守正本。**

## 四、三類回報的判斷重點 + 處理 SOP

每類都附「特別要注意什麼」與「實際要改哪些檔案、跑哪些工具」。工具流程一律遵循 `CLAUDE.md`。

### A. 資料更正

**A1. 關係連結錯誤（例：不是兄弟，是雙胞胎）**

- 判斷重點：雙胞胎＝同父母**且同生日**。先用 `python tools/query.py profile <名>` 比對兩隻的 `born` 是否相同、父母是否一致。相同 → 綠燈；只是同年不同日 → 只是手足，退回更正者。
- 改哪裡：兩隻條目的 `## 家族關係` 區塊文字（標明「雙胞胎」）。家系圖會依資料呈現雙胞胎。
- 跑：`build_db.py` → `export_json.py`。

**A2. 名字錯誤**

- 判斷重點：中文名／暱稱**以作者校訂為準**，回報只當參考（綠燈需有來源）。RPF 羅馬拼音只是後備。Other Names 記得填進 `japanese` / `english_variants` / `nicknames`。
- 改哪裡：該條目 frontmatter 對應欄位 + `## 名稱與別名` 表。
- 跑：`build_db.py` → `export_json.py`。

**A3. 生日錯誤** ⚠️ 連鎖效應最大

- 判斷重點：生日是 slug 的一部分。改生日 = **要改檔名**（`名字-新生日.md`）**並修正所有指向它的 `[[wikilink]]`**（見 `CLAUDE.md`「檔名與消歧」）。務必要有可靠來源才動，否則維持黃燈。
- 改哪裡：frontmatter `born:` → 重新命名檔案 → 全庫搜尋舊 slug 換成新 slug → 若 `zoos:` 用了年份起訖也檢查。
- 跑：`gen_residence.py` → `build_db.py` → `export_json.py`。

### B. 資料變更（轉園）

- 判斷重點：要有生效日期與來源（官網／公告）。新園若 `data/zoos.json` 沒登記，**先在註冊表加一筆**（canonical 日文名／中文／座標／官網），否則 `build_db.py` 會報錯中止。
- 改哪裡：該條目 frontmatter `zoos:`，把舊園的訖日填上、加一行新園 `(起 – 現在)`。**不要手改內文居住史表格**（自動生成）。
- 跑：`gen_residence.py`（重生居住史表）→ `build_db.py` → `export_json.py`。

### C. 出生 / 過世

**C1. 新生兒**（走 `CLAUDE.md`「新增成員流程」）

- 判斷重點：要有來源；確認父母條目是否已存在。
- 動作：建立 `wiki/名字-生日.md`（未命名先用佔位名＋媽媽名）→ **自動補齊直系親屬**（父、母、雙胞胎、子女）→ 回頭把相關條目親屬改成 `[[wikilink]]` → 更新 `index.md`（分類＋總數）→ `log.md` append。
- 跑：`build_db.py` → `audit.py`（檢查完整度）→ `export_json.py`。

**C2. 過世**

- 判斷重點：訃聞需有來源（官網／可靠社群）。
- 動作：frontmatter 加 `died: YYYY-MM-DD`、`tags` 加 `deceased`；標題與引言名稱後加 🌈；`index.md`／相關條目的 wikilink 名稱後加 🌈。
- 跑：`build_db.py` → `export_json.py`。（首頁「前往小熊星球」會自動反映。）

> 每類處理完都在 `wiki/log.md` 末端 append 一筆（`update` 或 `add`），來源寫回報依據的官網／RPF 連結。**log 內名字用 backtick，絕不用 `[[wikilink]]`。**

## 五、Claude 在這套流程中扮演什麼角色（我的建議）

之前那題問得太抽象，這裡用三個具體做法讓你選。差別只在「我幫你做到哪一步」：

**做法一（推薦）：我起草，你拍板。**
我把每筆回報跑紅綠燈、核對 RPF／lineage／官網，然後**直接把要改的 wiki 內容和 log 草擬好**給你看，你只需回「同意」或「退回」。我再執行寫入與重建。
→ 最省你的力氣，又 100% 符合資料主權（你仍是最後一關）。

**做法二：我全自動寫低風險的、高風險才問你。**
像「補一個空白欄位」「官網已公告的轉園」這類綠燈，我直接寫入並記 log；只有黃／紅燈才停下來問你。
→ 最快，但你的逐筆掌控度較低，偶爾可能需要事後回看。

**做法三：我只整理清單，wiki 全由你手改。**
我把收件匣彙整成分級待辦清單交給你，所有 `wiki/*.md` 都你親手改。
→ 你掌控度最高，但也最花你的時間。

我建議**做法一**：它在「省力」與「你完全掌控」之間最平衡，也最貼合你 CLAUDE.md 寫的「作者校訂後才寫入」。

## 七、風險與注意事項

- **灌水／惡意**：紅綠燈的紅燈就是防線；Tally/Google 可開驗證碼。無來源一律不動正本。
- **隱私**：表單別強制要 email；致謝用暱稱即可。
- **生日改名連鎖**（A3）：最容易出錯，務必改 slug 同時全庫修 wikilink，再重建。
- **新園未登記**：轉園前先補 `data/zoos.json`，否則 build 中止。
- **條目總數**：更新後用 `ls wiki/*.md | wc -l` 減 2 驗證，別只信 index 頁首數字。
